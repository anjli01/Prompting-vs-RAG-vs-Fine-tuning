    02 / 09

    🗺️ The Mental Model — 3 Layers

    LLM Customization as complementary layers, not competing choices

    🎯

    Prompting

    THE INSTRUCTION LAYER

    "How you talk to the model"

    Nothing changes — you just ask better

    📚

    RAG

    THE KNOWLEDGE LAYER

    "What you show the model"

    Give it your data at query time

    🔧

    Fine-Tuning

    THE BEHAVIOR LAYER

    "How you train the model"

    The model itself permanently changes

    🍳 The Chef Analogy

    🗣️

    Prompting

    Giving the chef verbal instructions
    "Make it spicy, give me bullet points"

    Same chef, same knowledge

    📖

    RAG

    Handing the chef a recipe book
    "Here's our company data, refer to it"

    New information at cook-time

    🎓

    Fine-Tuning

    Sending the chef to culinary school
    "Learn a new cuisine permanently"

    The chef itself changes

    03 / 09

    🧭 When to Use Which?

    Ask yourself these 3 simple questions

    ❓

    Q1: Does it need data it doesn't have?

    Company docs, policies, live data, anything the model was never trained on

    → Use RAG 📚

    ❓

    Q2: Is the output format or reasoning bad?

    Messy structure, wrong logic, no step-by-step thinking

    → Better Prompting 🎯

    ❓

    Q3: Good prompts but still inconsistent at scale?

    Works sometimes, breaks sometimes, need it reliable for 1000s of requests

    → Fine-Tune 🔧

    🪜 The Golden Rule — Always start simple, escalate only when needed

    Step 1: Prompting
    Always start here
    →
    Step 2: Add RAG
    If needs external data
    →
    Step 3: Fine-Tune
    Last resort only
    →
    🏆 Combine All
    Production systems
    04 / 09

    ⚠️ Common Mistakes to Avoid

    🚨 The #1 Mistake

    "The model doesn't know our company data"
"Let's fine-tune it on our documents!"

    ✅ The Right Answer

    This is a KNOWLEDGE problem → Use RAG
Fine-tuning teaches STYLE, not FACTS
It teaches HOW to respond, not WHAT to say

    🎯 Prompting

    ❌
    Being too vague

    "Summarize this" → Be specific: "3 bullets, max 20 words, focus on finances"

    ❌
    No output format specified

    Hoping the model guesses → Always say "respond in JSON" or "use bullet points"

    📚 RAG

    ❌
    Dumping too much context

    20+ chunks → Model ignores the middle. Use top 3-5 relevant ones.

    ❌
    Bad chunking strategy

    Blindly splitting at 512 tokens → Use semantic/meaningful boundaries

    🔧 Fine-Tuning

    🚨
    Using it to teach facts

    Facts = RAG. Fine-tuning is for consistent style/format only.

    ❌
    Skipping prompting first

    Always try prompt + RAG before fine-tuning. Most problems are solved without it.

    05 / 09

    🎯 Prompting — The Instruction Layer

    "How you talk to the model" — always start here

    TechniqueWhat It MeansWhen to Use
    Zero-shotJust ask directly, no examplesSimple questions, quick tasks
    Few-shotGive 2-3 examples first, then askNeed consistent output format
    Chain-of-ThoughtTell model to "think step by step"Complex reasoning, multi-step problems
    System PromptSet a role/persona for the modelConsistent behavior across conversations
    Example 
    # Zero-shot — just ask (simple but unpredictable format)
"Classify this email: 'The AC on floor 5 is broken'"
# Output: "This is a Maintenance request"  — works but format varies

# Few-shot — give examples first (consistent output format)
"Examples:
  'Rent failed' → BILLING | 95%
  'Elevator stuck' → MAINTENANCE | 98%
  Classify: 'AC broken on floor 5'"
# Output: MAINTENANCE | 96%  — consistent, parseable every time

    ✅ Pros

    Free • Instant • No setup needed • Works with any model

    ❌ Cons

    Can't inject private data • Output can be inconsistent at scale

    06 / 09

    📚 RAG — The Knowledge Layer

    "What you show the model" — give it your data at query time

    📥 Phase 1: Setup (Do once)

    📄 Your Documents (PDFs, Policies)
    ↓ Split into pieces
    ✂️ Chunks of text
    ↓ Convert to numbers
    🗄️ Store in Vector Database

    🔍 Phase 2: Every Query (Real-time)

    ❓ User asks a question
    ↓ Find relevant chunks
    📋 Top 3-5 matching chunks
    ↓ Send to LLM with question
    ✅ Grounded answer from your data

    ✅ Pros

    Uses your private data • Always up-to-date • Reduces hallucination • No model retraining

    ❌ Cons

    Needs vector DB setup • Answer quality depends on chunk quality • Slightly slower

    07 / 09

    🔧 Fine-Tuning — The Behavior Layer

    "How you train the model" — last resort, for permanent behavior change

    How It Works

    1

    Create training examples (JSONL)

    100–500+ examples of ideal input/output pairs

    2

    Run a training job

    Upload to AWS Bedrock / Azure / Google and train

    3

    Get a new model that behaves differently

    Same API, new model ID — permanently changed

    Methods

    MethodEffortBest For
    Full Fine-Tune🔴 Very HighRarely used in practice
    LoRA🟡 ModerateMost practical approach
    QLoRA🟢 LowWorks on consumer GPUs

    Training Data Example

    JSONL 
    // Each line teaches one behavior
{"messages": [
  {"role": "system", "content": "Respond in JSON."},
  {"role": "user", "content": "AC broken floor 5"},
  {"role": "assistant", "content":
    "{\"category\": \"MAINTENANCE\",
     \"priority\": \"HIGH\"}"}
]}

    ✅ Pros

    Consistent output every time • No long prompts needed • Fast inference

    ❌ Cons

    Needs lots of training data • Weeks to set up • Static — can't learn new facts

    08 / 09

    🏗️ Hybrid Architecture — All Three Combined

    How production systems actually work — all three layers stacked together

    How They Stack

    👤

    User Query

    "What's the occupancy trend for Tower B?"

    🎯

    Layer 1: Prompting

    System prompt → Analyst role, JSON format, no-speculation rule

    📚

    Layer 2: RAG

    Retrieve lease docs, occupancy reports, market rates → Top 5 chunks

    🔧

    Layer 3: Fine-Tuned Model

    Model trained on your report format, risk scoring, professional tone

    ✅

    Grounded Output

    Consistent, factual, well-formatted answer every time

    What Each Layer Adds

    LayerRoleWithout It
    🎯 PromptControls reasoning & formatInconsistent responses
    📚 RAGInjects real-time dataHallucinated facts
    🔧 Fine-TuneConsistent style & schemaComplex prompts needed

    💡 Key Takeaway

    These are not alternatives — they are layers that stack. Remove any one layer and something breaks.

    09 / 09

    💻 Code Example — All Three in Action

    Same use case, building on each layer — Prompting → RAG → Fine-Tuning

    Python 
    import anthropic, chromadb
client = anthropic.Anthropic()

question = "What is the occupancy trend for Tower B?"

# ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
# STEP 1: PROMPTING ONLY — just talk to the model
# ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
response = client.messages.create(
    model="claude-sonnet-4-20250514",
    system="You are a property analyst. Respond in JSON with: summary, trend, recommendation.",
    messages=[{"role": "user", "content": question}]
)
# ⚠️ Works but model has NO real data — may hallucinate facts about Tower B


# ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
# STEP 2: ADD RAG — give it real data before asking
# ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
collection = chromadb.Client().get_collection("property_docs")
results = collection.query(query_texts=[question], n_results=5)
context = "\n".join(results["documents"][0])  # top 5 relevant chunks

response = client.messages.create(
    model="claude-sonnet-4-20250514",
    system="You are a property analyst. Use ONLY the provided context. Respond in JSON.",
    messages=[{"role": "user", "content": f"Context:\n{context}\n\nQuestion: {question}"}]
)
# ✅ Now grounded in real docs — but output format may still vary


# ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
# STEP 3: ADD FINE-TUNING — swap in your trained model
# ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
response = client.messages.create(
    model="claude-haiku-property-v1",  # ← fine-tuned model, same API
    system="You are a property analyst. Use ONLY the provided context.",
    messages=[{"role": "user", "content": f"Context:\n{context}\n\nQuestion: {question}"}]
)
# 🏆 Grounded + consistent format + professional tone — every single time

    Step 1: Prompting

    Controls reasoning & format

    + Step 2: RAG

    Adds real data & facts

    + Step 3: Fine-Tune

    Consistent behavior at scale
